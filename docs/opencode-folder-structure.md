# OpenCode AI Tool Configuration

This document describes the `.opencode/` folder structure used by the OpenCode AI coding agent.

> **Note:** This folder is **gitignored** following the upstream OpenSpec practice. It contains generated files specific to your local development environment.

## Overview

The `.opencode/` directory is generated by `openspec init --tools opencode`. It contains:
- **Skills** - AI agent skills for OpenSpec workflow commands
- **Commands** - Slash commands for invoking skills
- **node_modules** - Dependencies (not tracked in git)

## Directory Structure

```
.opencode/
├── skills/                          # AI agent skills
│   ├── openspec-explore/
│   │   └── SKILL.md                 # Explore mode skill
│   ├── openspec-propose/
│   │   └── SKILL.md                 # Propose change skill
│   ├── openspec-apply-change/
│   │   └── SKILL.md                 # Apply tasks skill
│   ├── openspec-verify-change/
│   │   └── SKILL.md                 # Verify implementation skill
│   ├── openspec-sync-specs/
│   │   └── SKILL.md                 # Sync specs skill
│   └── openspec-archive-change/
│       └── SKILL.md                 # Archive change skill
├── commands/                        # Slash commands (newer format)
│   ├── explore.md
│   ├── propose.md
│   ├── apply.md
│   └── archive.md
├── command/                         # Slash commands (legacy format)
│   ├── opsx-explore.md
│   ├── opsx-propose.md
│   ├── opsx-apply.md
│   └── opsx-archive.md
├── .gitignore                       # Nested gitignore
├── package.json                     # Dependencies manifest
├── bun.lock                         # Lock file
└── node_modules/                    # Dependencies (gitignored)
```

## Skills

Each skill is a `SKILL.md` file with YAML frontmatter and markdown instructions:

```yaml
---
name: openspec-apply-change
description: Implement tasks from an OpenSpec change.
license: MIT
compatibility: Requires openspec CLI.
metadata:
  author: OpenSpec
  version: "1.0"
  generatedBy: "OpenSpec v0.1.3"
---

Implement tasks from an OpenSpec change...

**Steps**
1. Select the change
2. Check status
3. Get apply instructions
...
```

### Available Skills

| Skill | Purpose |
|-------|---------|
| `openspec-explore` | Thinking partner for exploring ideas |
| `openspec-propose` | Create change with proposal, specs, design, tasks |
| `openspec-apply-change` | Implement tasks from a change |
| `openspec-verify-change` | Verify implementation matches artifacts |
| `openspec-sync-specs` | Sync delta specs to main specs |
| `openspec-archive-change` | Archive completed change |

## Git Ignore

The `.opencode/` folder is **gitignored** at the project level:

```gitignore
/.opencode
```

This follows the upstream OpenSpec practice. The skills are generated from embedded templates in the binary, so they don't need to be versioned.

### Nested .gitignore

Inside `.opencode/` there's a nested `.gitignore`:

```gitignore
node_modules
package.json
bun.lock
.gitignore
```

This excludes package manager files even if someone accidentally removes the top-level gitignore.

## How It's Generated

```bash
# Initialize with OpenCode support
openspec init --tools opencode

# Or update after OpenSpec upgrade
openspec update .
```

The binary:
1. Reads embedded skill templates from `templates/skills/*.md`
2. Generates `SKILL.md` files with version metadata
3. Creates command files for each skill
4. Sets up nested gitignore

## Team Consistency

Since `.opencode/` is gitignored, how do multiple developers on the same project get consistent AI capabilities?

**The key insight**: Skills and commands are generated from templates **embedded in the OpenSpec binary**, not from files in the repository.

### How It Works

1. **Same binary version** → Same embedded templates → Identical output
2. Each developer runs `openspec init --tools opencode` (or `openspec update .`)
3. OpenSpec generates the `.opencode/` folder from its embedded templates
4. All team members get identical skill/command files

This is analogous to `node_modules/`:
- Not committed to git
- Reproducible via the tool (npm install → openspec init)
- Version-controlled by the tool version (package.json → OpenSpec binary version)

### Ensuring Consistency

To ensure all developers have the same OpenSpec version:

1. **Document the version** in README or CONTRIBUTING.md
2. **Pin the version** in install instructions (e.g., `curl -L https://.../v0.1.3/openspec-linux-x64`)
3. **Run `openspec update .`** after pulling changes that use new OpenSpec features

## How Slash Commands Work

OpenCode discovers and loads commands from two directories:

### `command/` Folder (Legacy Format)

Files named `opsx-<name>.md` become slash commands:

| File | Command |
|------|---------|
| `command/opsx-apply.md` | `/opsx-apply` |
| `command/opsx-explore.md` | `/opsx-explore` |
| `command/opsx-propose.md` | `/opsx-propose` |

### `commands/` Folder (Newer Format)

Files named `<name>.md` become namespaced slash commands:

| File | Command |
|------|---------|
| `commands/apply.md` | `/opsx:apply` |
| `commands/explore.md` | `/opsx:explore` |
| `commands/propose.md` | `/opsx:propose` |

OpenCode prefixes project commands with `opsx:` to avoid conflicts with built-in commands.

### Command Execution

When you type a slash command (e.g., `/opsx-apply add-auth`):

1. OpenCode matches the command to a file in `command/` or `commands/`
2. Reads the markdown content
3. Injects it as instructions to the AI agent
4. The AI executes the workflow defined in that markdown

The markdown file acts like a "prompt template" that guides the AI through a specific workflow.

## Regeneration

To regenerate after updating OpenSpec:

```bash
openspec update . --force
```

This overwrites existing files with fresh templates from the binary.

## Other AI Tools

OpenSpec supports multiple AI tools. Each has its own directory:

| Tool | Directory | Gitignored |
|------|-----------|------------|
| OpenCode | `.opencode/` | Yes |
| Claude | `.claude/` | Yes |
| Cursor | `.cursor/` | Yes |
| Windsurf | `.windsurf/` | Yes |
| Codex | `.codex/` | Yes |

Initialize multiple tools:
```bash
openspec init --tools opencode,claude,cursor
```
